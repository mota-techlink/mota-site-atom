# Template Fork Workflow (Plan A)

This repository is positioned as a "Feature Template Repo", used to incubate features, fix issues, and then sync to production projects via fork.

## Goals

- Stable iteration of feature code: placed in `src/`
- Content can be independently replaced: placed in `content/` + `public/`
- Multi-language content maintained independently: placed in `content/locales/`

## Directory Conventions (Currently Implemented)

- `content/site/i18n.yml`: Single source of truth for language configuration
- `content/locales/*.json`: Single source of truth for multi-language content
- `content/site/nav.yml`: Navigation structure configuration
- `content/site/docs-nav.yml`: Documentation sidebar structure
- `content/site/features.yml`: Homepage Features structure
- `content/site/site.yml`: Site branding and public display information
- `content/site/menu.yml`: Dashboard/Admin menu structure configuration

Validation scripts automatically generate runtime artifacts to `src/generated/content-config/*.json` (do not edit manually).

`src/config/*` now serves as an "adapter layer" and no longer stores core content data.

## Recommended Branch Strategy

### Template Repository (This Repo)

- `main`: Template stable branch (can be synced by production projects)
- `feature/*`: New feature development
- `fix/*`: Bug fixes

### Production Project (After Fork)

- Keep upstream pointing to the template repository
- Periodically sync template features without overwriting own content

```bash
git remote add upstream https://github.com/mota-techlink/mota-atom.git
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

## Recommended Branch Workflow (Production Project)

Keep feature development and content maintenance in **separate branches**, unified through a dedicated release branch for deployment.

### Branch Overview

| Branch | Purpose |
|--------|---------|
| `main` | Stays in sync with upstream, no direct development |
| `feature` | Code changes (`src/`, `scripts/`, etc.) |
| `content` | Content and brand changes (`content/`, `public/`) |
| `release` | Deployment branch — merges feature + content |

### Workflow Diagram

```
upstream/main
     │
     ▼  bash scripts/sync-upstream.sh
   main ──────────────► feature ──┐
                                   ├──► release (deploy)
                       content ───┘
                   bash scripts/release.sh
```

### Day-to-Day Operations

**Feature development (feature branch):**

```bash
git checkout feature
# Only modify src/, scripts/, etc.
git add src/ scripts/
git commit -m "feat: xxx"
```

**Content updates (content branch):**

```bash
git checkout content
# Only modify content/, public/
git add content/ public/
git commit -m "content: xxx"
```

**Sync upstream (one command):**

```bash
bash scripts/sync-upstream.sh
# Auto: fetch upstream → protect content/public → sync to main → merge to feature → push
```

**Release (one command):**

```bash
bash scripts/release.sh
# Auto: feature + content → merge to release → push
```

## Content Replacement Process (Production Project)

1. Replace business content under `content/` (docs, products, multi-language)
2. Replace brand assets under `public/` (logos, images, videos)
3. Keep `src/` feature layer, avoid divergence conflicts with template features

## Syncing Only Code Changes (Excluding content/ and public/)

The upstream template repository for this project is:

```
https://github.com/mota-techlink/mota-atom.git
```

If you only want to sync feature code from upstream without bringing in upstream changes to `content/` and `public/`, the following approaches are available:

### Option 1: gitattributes Content Protection (with merge)

> ⚠️ **Known Limitation**: `merge=ours` only applies to files that **already exist locally**. If upstream adds new files under `content/` or `public/` (`create mode`), those files **will still be merged in** and cannot be blocked by this approach. Use this only when upstream modifies existing files.

Once `.gitattributes` is configured (see the "Best Practice" section below), a standard merge will automatically protect your content directories:

```bash
git fetch upstream
git merge upstream/main
# Changes in content/ and public/ are automatically ignored; src/ code merges normally
git push origin main
```

### Option 2: Selective Path Sync

Precisely control what gets synced by checking out only code-related files from upstream, leaving `content/` and `public/` completely untouched:

```bash
git fetch upstream
# Sync only code and configuration paths
git checkout upstream/main -- src/ scripts/ package.json tsconfig.json \
  next.config.ts tailwind.config.ts postcss.config.mjs \
  eslint.config.mjs webpack.config.js
git commit -m "Sync code from upstream/main, local content unchanged"
git push origin main
```

> **Note**: This approach directly replaces local files with upstream content without creating a merge commit. It gives precise control over the sync scope but does not automatically handle new files added to upstream.

### Option 3: Merge then Reset Content Directories (Recommended)

Merge all upstream changes, then force-restore your local content. **This fully blocks all upstream changes to `content/` and `public/`, including newly added files.**

```bash
git fetch upstream
git merge upstream/main --no-commit --no-ff

# Force-restore local content/ and public/ (overrides upstream modifications to existing files)
git checkout HEAD -- content/ public/

# Remove any new files upstream added under content/ or public/
git status --short | grep '^A  content\|^A  public' \
  | awk '{print $2}' | xargs -r git rm --cached
git status --short | grep '^?? content\|^?? public' \
  | awk '{print $2}' | xargs -r rm -rf

git commit -m "sync: upstream code only, content and public preserved"
git push origin main
```

## Best Practice: Conflict Avoidance Strategy

### Core Pain Point

When the template repository (Repo A) updates feature code and also modifies example documentation or images in `content/`, the production project (Repo B) may encounter issues when running `git merge upstream/main`:
1. **Conflict**: Repo B has also modified the same file.
2. **Overwrite**: Files unmodified in Repo B are overwritten by new examples from Repo A.

### Solution: Git Attributes Content Protection (Recommended)

Define a merge driver to tell Git to **always prioritize the current branch (Repo B)** when merging `content` and `public` directories, automatically ignoring Upstream content changes.

**1. Configure Merge Driver in Production Project Root:**

This is a one-time local configuration:

```bash
git config merge.ours.driver true
```

**2. Create `.gitattributes` File in Production Project Root:**

Commit this file to the repository to ensure the strategy is shared among team members:

```text
content/**/* merge=ours
public/**/* merge=ours
```

**3. Verification**

After configuration, when you run `git merge upstream/main`:
- Code changes in `src/`: **Merged normally** (get new features).
- Content changes in `content/`: **Automatically discard Upstream changes**, keeping your own business data.

### Alternative: Manual Conflict Resolution

If you prefer not to configure `.gitattributes`, you can use the following commands to quickly keep your version when conflicts occur:

```bash
git merge upstream/main
# If content/ conflict occurs
git checkout --ours content/
git checkout --ours public/
git add .
git commit -m "Merge upstream features but keep local content"
```

## Conflict Minimization Tips (Daily Development)

- **Separate Structure**: Do not directly modify `src/config/*` structure in production projects; this directory should follow template updates.
- **Extract Text**: Move all business text to `content/locales/` to avoid hardcoding in components.
- **Centralize Config**: Concentrate navigation and site info changes in `content/site/*`.

## New Page Content Standards

For new pages (e.g., about/contact/future portal pages), follow these standards:

1. Create page configuration in `content/site/*.yml` (at minimum include `metadata.titleKey` and `metadata.descriptionKey`)
2. Page UI text should only reference translation keys, never hardcode text
3. Page metadata should use the unified helper: `src/lib/page-metadata.ts`
4. Translations should only be written to `content/locales/*.json`
5. Run `npm run content:validate` before committing to ensure schema and translation keys pass

## Pre-commit Protection (Built-in)

- Content Schema validation: `npm run content:validate`
- Template check: `npm run check:template`

To enable automatic pre-commit validation:

```bash
npm run hooks:install
```

Once enabled, each commit will automatically run `content:validate`, catching:

- Missing locale files
- Inconsistent locale keys
- `content/site/*` configuration structure errors

- The build pipeline uses `node scripts/*.mjs` to execute key scripts directly, avoiding the uncertainty of nested `npm run` calls.
- Recommended Pages Build command: `npm run pages:build`
- For data manifest generation only: `npm run build:data`
