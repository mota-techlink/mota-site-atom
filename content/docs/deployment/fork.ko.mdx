# Template Fork Workflow (플랜 A)

본 리포지토리는 "기능 템플릿 리포지토리(Template Repo)"로 위치하며, 기능 인큐베이션, 이슈 수정 후 fork를 통해 프로덕션 프로젝트에 동기화합니다.

## 목표

- 기능 코드의 안정적인 반복: `src/`에 배치
- 콘텐츠 독립 교체 가능: `content/` + `public/`에 배치
- 다국어 콘텐츠 독립 유지보수: `content/locales/`에 배치

## 디렉토리 규약 (현재 구현 완료)

- `content/site/i18n.yml`: 언어 설정의 유일한 진실 공급원
- `content/locales/*.json`: 다국어 텍스트의 유일한 진실 공급원
- `content/site/nav.yml`: 네비게이션 구조 설정
- `content/site/docs-nav.yml`: 문서 사이드바 구조
- `content/site/features.yml`: 홈페이지 Features 구조
- `content/site/site.yml`: 사이트 브랜드 및 공개 표시 정보
- `content/site/menu.yml`: Dashboard/Admin 메뉴 구조 설정

검증 스크립트는 자동으로 런타임 아티팩트를 `src/generated/content-config/*.json`에 생성합니다 (수동 편집 금지).

`src/config/*`는 현재 "어댑터 레이어"로 기능하며, 더 이상 핵심 콘텐츠 데이터를 저장하지 않습니다.

## 권장 브랜치 전략

### 템플릿 리포지토리 (본 리포지토리)

- `main`: 템플릿 안정 브랜치 (각 프로덕션 프로젝트가 동기화 가능)
- `feature/*`: 새 기능 개발
- `fix/*`: 버그 수정

### 프로덕션 프로젝트 (fork 후)

- upstream을 템플릿 리포지토리로 유지
- 정기적으로 템플릿 기능을 동기화하되, 자체 콘텐츠를 덮어쓰지 않음

```bash
git remote add upstream https://github.com/mota-techlink/mota-atom.git
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

## 콘텐츠 교체 프로세스 (프로덕션 프로젝트)

1. `content/` 하위 비즈니스 콘텐츠 교체 (문서, 제품, 다국어)
2. `public/` 하위 브랜드 자산 교체 (로고, 이미지, 비디오)
3. `src/` 기능 레이어 유지, 템플릿 기능과의 분기 충돌 방지

## 코드 변경 사항만 동기화 (content/ 및 public/ 제외)

본 프로젝트의 업스트림 템플릿 리포지토리는 다음과 같습니다:

```
https://github.com/mota-techlink/mota-atom.git
```

업스트림에서 기능 코드만 동기화하고 `content/` 및 `public/` 변경 사항은 가져오지 않으려면 다음 방법을 사용할 수 있습니다:

### 방법 1: gitattributes 콘텐츠 보호 (merge와 함께, 권장)

`.gitattributes` 설정 완료 후 (아래 "모범 사례" 섹션 참조), 일반 merge를 실행하면 콘텐츠 디렉터리가 자동으로 보호됩니다:

```bash
git fetch upstream
git merge upstream/main
# content/ 및 public/ 변경 사항은 자동으로 무시되고, src/ 등의 코드는 정상 병합됩니다
git push origin main
```

### 방법 2: 선택적 경로 동기화

동기화 범위를 정밀하게 제어하여 업스트림에서 코드 관련 파일만 체크아웃하고 `content/` 와 `public/` 는 완전히 건드리지 않습니다:

```bash
git fetch upstream
# 코드 및 설정 관련 경로만 동기화
git checkout upstream/main -- src/ scripts/ package.json tsconfig.json \
  next.config.ts tailwind.config.ts postcss.config.mjs \
  eslint.config.mjs webpack.config.js
git commit -m "Sync code from upstream/main, local content unchanged"
git push origin main
```

> **참고**: 이 방법은 병합 커밋을 생성하지 않고 업스트림 파일 내용으로 로컬 파일을 직접 교체합니다. 동기화 범위를 정확히 제어할 수 있지만 업스트림에 추가된 새 파일은 자동으로 처리되지 않습니다.

### 방법 3: 병합 후 콘텐츠 디렉터리 리셋

업스트림의 모든 변경 사항을 병합한 후 로컬 콘텐츠를 강제 복원합니다:

```bash
git fetch upstream
git merge upstream/main --no-commit --no-ff
# 병합 스테이징 후 로컬 콘텐츠 디렉터리 강제 복원
git checkout HEAD -- content/ public/
git commit -m "Sync upstream code changes, keep local content"
git push origin main
```

## 모범 사례: 충돌 방지 전략

### 핵심 문제점

템플릿 리포지토리(Repo A)가 기능 코드를 업데이트하면서 동시에 `content/` 하위의 예제 문서나 이미지를 수정했을 때, 프로덕션 프로젝트(Repo B)에서 `git merge upstream/main`을 실행하면 다음과 같은 문제가 발생할 수 있습니다:
1. **충돌 (Conflict)**: Repo B도 동일한 파일을 수정한 경우.
2. **덮어쓰기 (Overwrite)**: Repo B에서 수정하지 않은 파일이 Repo A의 새 예제로 덮어씌워지는 경우.

### 해결책: Git Attributes 콘텐츠 보호 (권장)

merge driver를 정의하여 Git에게 `content` 및 `public` 디렉터리를 병합할 때 **항상 현재 브랜치(Repo B)를 우선**하도록 지시하여 Upstream의 콘텐츠 변경을 자동으로 무시하게 합니다.

**1. 프로덕션 프로젝트 루트에서 Merge Driver 설정:**

이것은 일회성 설정입니다 (로컬 적용):

```bash
git config merge.ours.driver true
```

**2. 프로덕션 프로젝트 루트에 `.gitattributes` 파일 생성:**

이 파일을 리포지토리에 커밋하여 팀원들이 이 전략을 공유하도록 하세요:

```text
content/**/* merge=ours
public/**/* merge=ours
```

**3. 효과 확인**

설정 후 `git merge upstream/main` 실행 시:
- `src/` 하위 코드 변경: **정상 병합** (새 기능 획득).
- `content/` 하위 콘텐츠 변경: **Upstream 변경 자동 폐기**, 자체 비즈니스 데이터 유지.

### 대안: 수동 충돌 해결

`.gitattributes`를 설정하고 싶지 않은 경우, 충돌 발생 시 다음 명령어를 사용하여 빠르게 내 버전을 유지할 수 있습니다:

```bash
git merge upstream/main
# content/ 충돌 발생 시
git checkout --ours content/
git checkout --ours public/
git add .
git commit -m "Merge upstream features but keep local content"
```

## 충돌 최소화 팁 (일상 개발)

- **구조 분리**: 프로덕션 프로젝트에서 `src/config/*` 구조를 직접 수정하지 마세요. 이 디렉터리는 템플릿 업데이트를 따라야 합니다.
- **텍스트 추출**: 모든 비즈니스 텍스트를 `content/locales/`로 이동하여 컴포넌트 내 하드코딩을 피하세요.
- **설정 집중**: 네비게이션 및 사이트 정보 변경을 `content/site/*`에 집중하세요.

## 새 페이지 콘텐츠 규약

새 페이지 (예: about/contact/향후 포털 페이지)는 다음 규약을 통일적으로 준수하세요:

1. `content/site/*.yml`에 페이지 설정 생성 (최소한 `metadata.titleKey`와 `metadata.descriptionKey` 포함)
2. 페이지 UI 텍스트는 번역 키만 참조하고, 텍스트를 하드코딩하지 않기
3. 페이지 메타데이터는 통합 헬퍼 사용: `src/lib/page-metadata.ts`
4. 번역은 `content/locales/*.json`에만 작성
5. 커밋 전 `npm run content:validate` 실행하여 스키마 및 번역 키 검증 확인

## 커밋 전 보호 (내장)

- 콘텐츠 스키마 검증: `npm run content:validate`
- 템플릿 검사: `npm run check:template`

Git 커밋 전 자동 검증(pre-commit)을 활성화하려면:

```bash
npm run hooks:install
```

활성화하면 각 커밋 시 자동으로 `content:validate`가 실행되어 다음을 우선 감지합니다:

- 누락된 locale 파일
- 불일치하는 locale 키
- `content/site/*` 설정 구조 오류

## Cloudflare Pages 배포 참고사항

- 빌드 파이프라인은 `node scripts/*.mjs`를 사용하여 핵심 스크립트를 직접 실행하며, 중첩된 `npm run` 호출로 인한 불확실성을 방지합니다.
- 권장 Pages Build 명령어: `npm run pages:build`
- 데이터 매니페스트 생성만 필요한 경우: `npm run build:data`
