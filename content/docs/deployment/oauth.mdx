## Google Side
1. Create a Google Cloud project: Visit the [Google Cloud Console](https://console.cloud.google.com/) to create a new project.

2. Configure the OAuth consent screen: In "APIs & Services" -> "OAuth consent screen", select "External" and fill in the necessary information.

3. Create [credentials](https://console.cloud.google.com/apis/credentials): On the "Credentials" page, click "Create credentials" -> "OAuth client ID".
![Create OAuth Client ID](/images/docs/CreateOAuthClientID.webp)

- Select "Web application" as the application type.

- Authorized JavaScript origins: Add `http://localhost:3000`.

- Authorized redirect URIs: Add the local Supabase callback URL, in the format `http://127.0.0.1:54321/auth/v1/callback`.

4. Get Client ID and Secret: After successful creation, you will get a Client ID and Client Secret.

5. Configure local Supabase: Open the local Supabase Studio (`http://127.0.0.1:54323`), go to "Authentication" -> "Providers", enable "Google", and enter the Client ID and Client Secret you just obtained.


## Local Supabase Setup

When using Supabase locally, third-party login providers (such as Google, GitHub, etc.) in Authentication cannot be enabled directly in the local Studio UI.

You need to enable them by editing the Supabase configuration file in your project.

Here are the detailed steps to enable local Supabase Auth Providers:

**Core Step: Modify the config.toml file**
All configurations for local Supabase are located in the `supabase/config.toml` file in your project's root directory. You need to find or add the corresponding OAuth provider configuration block in this file and fill in your Client ID and Secret.

1. Find the configuration file
Open your project and find the following file: `Your project root directory/supabase/config.toml`

2. Get credentials from the third-party provider
Before configuring, you must first go to the third-party service provider you want to enable (such as Google Cloud Console, GitHub Developer Settings, etc.) to register an application and obtain the following two key pieces of information:

- Client ID

- Client Secret

Important note for local development: When configuring the "Authorized redirect URIs" of the third-party application, the default callback address for local Supabase is usually: `http://127.0.0.1:54321/auth/v1/callback` (please make sure the port number is consistent with the API URL port displayed when you start supabase)

3. Edit config.toml
In the `config.toml` file, scroll down to find the `[auth.external]` related area. If it doesn't exist, you need to add it manually.

Example: Enable Google login

Add or modify the following content in the file, replacing `your_client_id` and `your_client_secret` with the values you actually obtained:

```Ini, TOML
# supabase/config.toml

# ... other configurations ...

[auth.external.google]
enabled = true
client_id = "Your Google Client ID"
secret = "Your Google Client Secret"
# redirect_uri usually does not need to be set explicitly locally, Supabase will handle it automatically,
# but make sure you have configured the correct localhost callback address in the Google console.

# Example: Enable GitHub login
# [auth.external.github]
# enabled = true
# client_id = "Your GitHub Client ID"
# secret = "Your GitHub Client Secret"

# ... other configurations ...
```
Make sure `enabled = true`.

4. Restart local Supabase
After modifying the configuration file, you must restart the local Supabase service for the changes to take effect.

Run in your terminal:

```Bash
npx supabase stop
npx supabase start
```
5. Verification
After restarting, visit the local Supabase Studio panel again (usually `http://127.0.0.1:54323`).

Go to the Authentication -> Providers page. You should now see that the provider you just added in the configuration file (e.g., Google) is already displayed as "Enabled", and you can expand it to view the Client ID you configured.
