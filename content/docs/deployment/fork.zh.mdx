# Template Fork Workflow (方案A)

本仓库定位为“功能模板仓库（Template Repo）”，用于孵化功能、修复问题，再通过 fork 同步到正式项目。

## 目标

- 功能代码稳定迭代：放在 `src/`
- 内容可独立替换：放在 `content/` + `public/`
- 多语言内容独立维护：放在 `content/locales/`

## 目录约定（当前已落地）

- `content/site/i18n.yml`: 语言配置唯一真相源
- `content/locales/*.json`: 多语言文案唯一真相源
- `content/site/nav.yml`: 导航结构配置
- `content/site/docs-nav.yml`: 文档侧栏结构
- `content/site/features.yml`: 首页 Features 结构
- `content/site/site.yml`: 站点品牌与对外展示信息
- `content/site/menu.yml`: Dashboard/Admin 菜单结构配置

校验脚本会自动生成运行时产物到 `src/generated/content-config/*.json`（请勿手改）。

`src/config/*` 现在作为“适配层”，不再存放核心内容数据。

## 建议分支策略

### 模板仓库（本仓库）

- `main`: 模板稳定分支（可被各正式项目同步）
- `feature/*`: 新功能开发
- `fix/*`: 缺陷修复

### 正式项目（fork 后）

- 保持 upstream 指向模板仓库
- 定期同步模板功能，不覆盖自身内容

```bash
git remote add upstream https://github.com/mota-techlink/mota-atom.git
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

## 内容替换流程（正式项目）

1. 替换 `content/` 下业务内容（文档、产品、多语言）
2. 替换 `public/` 下品牌资源（logo、图、视频）
3. 保留 `src/` 功能层，避免与模板功能分叉冲突

## 只同步代码变更（排除 content/ 和 public/）

本项目上游模板仓库地址为：

```
https://github.com/mota-techlink/mota-atom.git
```

如果你只希望从上游同步功能代码，而不引入上游的 `content/` 和 `public/` 变更，有以下方案可选：

### 方案一：gitattributes 自动保护（配合 merge，推荐）

配置好 `.gitattributes` 后（见下方"最佳实践"一节），每次执行普通 merge 即可自动保护内容目录：

```bash
git fetch upstream
git merge upstream/main
# content/ 和 public/ 的变更将被自动忽略，src/ 等代码正常合并
git push origin main
```

### 方案二：选择性路径同步

精确控制同步范围，只从上游检出代码相关文件，完全不触碰 `content/` 和 `public/`：

```bash
git fetch upstream
# 只同步代码与配置相关路径
git checkout upstream/main -- src/ scripts/ package.json tsconfig.json \
  next.config.ts tailwind.config.ts postcss.config.mjs \
  eslint.config.mjs webpack.config.js
git commit -m "Sync code from upstream/main, local content unchanged"
git push origin main
```

> **注意**：此方式直接将上游文件内容替换本地文件，不产生合并提交。适合精确控制同步范围，但不会自动处理上游新增文件。

### 方案三：合并后重置内容目录

合并上游全部变更后，再强制恢复本地内容：

```bash
git fetch upstream
git merge upstream/main --no-commit --no-ff
# 合并暂存后强制恢复本地内容目录
git checkout HEAD -- content/ public/
git commit -m "Sync upstream code changes, keep local content"
git push origin main
```

## 最佳实践：规避冲突策略

### 核心痛点

当模板仓库（Repo A）更新了功能代码，同时也修改了 `content/` 下的示例文档或图片时，正式项目（Repo B）在执行 `git merge upstream/main` 时可能会遇到：
1. **冲突（Conflict）**：Repo B 也修改了同名文件。
2. **覆盖（Overwrite）**：Repo B 未修改的文件被 Repo A 的新示例覆盖。

### 解决方案：Git Attributes 内容保护（推荐）

通过定义 merge driver，告诉 Git 在合并 `content` 和 `public` 目录时，**永远以当前分支（Repo B）为准**，自动忽略 Upstream 的内容变动。

**1. 在正式项目根目录配置 Merge Driver：**

这是一个一次性配置（本地生效）：

```bash
git config merge.ours.driver true
```

**2. 在正式项目根目录新建 `.gitattributes` 文件：**

提交此文件到仓库，确保团队成员共享此策略：

```text
content/**/* merge=ours
public/**/* merge=ours
```

**3. 效果验证**

配置后，当你执行 `git merge upstream/main`：
- `src/` 下的代码变更：**正常合并**（获取新功能）。
- `content/` 下的内容变更：**自动丢弃 Upstream 变动**，保留你自己的业务数据。

### 备选方案：手动解决冲突

如果你不想配置 `.gitattributes`，在合并遇到冲突时，可以使用以下命令快速保留己方版本：

```bash
git merge upstream/main
# 如果提示 content/ 冲突
git checkout --ours content/
git checkout --ours public/
git add .
git commit -m "Merge upstream features but keep local content"
```

## 冲突最小化建议（日常开发）

- **结构分离**：不在正式项目里直接修改 `src/config/*` 结构，该目录应跟随模板更新。
- **文案抽离**：将业务文本全部移到 `content/locales/`，避免硬编码在组件中。
- **配置集中**：将导航和站点信息修改集中在 `content/site/*`。

## 新增页面内容规范

新增页面（如 about/contact/后续 portal 页面）请统一遵循：

1. 在 `content/site/*.yml` 建立页面配置（至少包含 `metadata.titleKey` 和 `metadata.descriptionKey`）
2. 页面 UI 文案只引用翻译 key，不写死文本
3. 页面 metadata 使用统一 helper：`src/lib/page-metadata.ts`
4. 翻译只写入 `content/locales/*.json`
5. 提交前执行 `npm run content:validate`，确保 schema 与翻译 key 都通过

## 提交前防护（已内置）

- 内容 Schema 校验：`npm run content:validate`
- 模板检查：`npm run check:template`

如需启用 git 提交前自动校验（pre-commit）：

```bash
npm run hooks:install
```

启用后，每次提交会自动执行 `content:validate`，优先拦截：

- 缺失 locale 文件
- locale key 不一致
- `content/site/*` 配置结构错误

## Cloudflare Pages 部署注意

- 构建链路已使用 `node scripts/*.mjs` 直接执行关键脚本，避免嵌套 `npm run` 带来的不确定性。
- 推荐 Pages Build command 使用：`npm run pages:build`
- 若只生成数据清单可使用：`npm run build:data`
